## Project Overview
A tool that monitors SSH authentication logs, detects brute-force login attacks (many failed logins from one IP in a short time window), and automatically responds with an escalating defense: alert -> rate-limit -> block -> auto-unblock after a configurable time. It has a web dashboard to visualize attacks and manage IP whitelist/blacklist.

## Critical requirement: Two modes
1. SIMULATION MODE (default, must work with zero external setup): a Python script generates realistic fake SSH auth log lines (mimicking /var/log/auth.log format on Ubuntu) representing both normal logins and brute-force attack patterns, written to a local log file (e.g. ./data/simulated_auth.log). The detector reads this file. No real firewall commands run in this mode - blocks are simulated/logged only.
2. LIVE MODE (optional, off by default): the same detector can be pointed at a real log file path (e.g. /var/log/auth.log) and will execute real `ufw` commands via subprocess to block IPs. Include hardcoded safety checks: never block 127.0.0.1/localhost, never block an IP the user has explicitly set as "admin IP" in config.

## Tech Stack
- Backend: Python 3.11+, FastAPI, SQLite (via SQLAlchemy), background asyncio task for log tailing
- Frontend: React (Vite) + Tailwind CSS + recharts for charts
- Deployment: Docker + docker-compose so the whole thing runs with one command

## Backend requirements
1. Log parser: regex-based parser for standard Linux SSH auth log lines (failed password, invalid user, accepted login), extracting timestamp, source IP, username, and result.
2. Log tailer: continuously reads new lines appended to the log file (handle the file growing, similar to `tail -f`), tracking last-read position so it doesn't reprocess old lines on restart.
3. Detection engine: sliding time-window tracker per source IP (e.g. dict of IP -> list of failed-attempt timestamps). Configurable threshold via a config file or .env (default: 5 failed attempts within 10 minutes = attack).
4. Whitelist/blacklist: stored in the database, checked before every detection/block decision. Provide CRUD API endpoints for managing these lists.
5. Escalation engine: when a threshold is crossed -> log an "alert" event. If attempts continue past a second higher threshold -> "rate-limit" event (simulated). If it continues further -> "block" event (real ufw command in Live Mode, simulated log entry in Simulation Mode). Auto-unblock after a configurable number of hours (default 24), with a background task that checks for expired blocks.
6. Alerting: log every event (alert/rate-limit/block/unblock/false-positive) to the database with full details (IP, username attempted, timestamp, action taken). Add an optional email alert function (SMTP config via .env) that's a no-op if not configured.
7. REST API (FastAPI) exposing: recent events/attacks feed, currently blocked IPs (+ manual unblock endpoint), whitelist/blacklist CRUD, current detection config, a "trigger simulated attack" endpoint that kicks off the log simulator for demo purposes, and a mode toggle (simulation/live) endpoint.
8. Use WebSockets (or Server-Sent Events) so the frontend can show new events in real time without polling.

## Frontend requirements
1. Single-page dashboard (React + Vite + Tailwind) with these sections:
   - Header showing current mode (Simulation/Live) with a toggle switch

2. Clean, modern security-dashboard aesthetic (dark theme, monospace font for IPs/logs, clear color coding: green=allowed, yellow=alert, orange=rate-limited, red=blocked)
3. Fully responsive
